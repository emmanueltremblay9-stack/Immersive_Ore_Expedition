# GeOre compatibility contract

## Final design target

The canonical target is [`BUDDING_FINAL_DECISIONS.md`](BUDDING_FINAL_DECISIONS.md). IOE must pre-register four distinct Budding blocks for every explicitly supported and validated GeOre material: `DAMAGED`, `CHIPPED`, `FLAWED`, and `FLAWLESS`.

Registration is permitted only when the GeOre resource, growth outputs, storage block, IOE profile, models, textures, recipes, and loot tables are all valid. A missing registry resource is skipped; IOE never invents a replacement.

Each IOE GeOre Budding family must reproduce the degradation, restoration, growth-face, update, Flawless, and compatible-accelerator behavior of the AE2 version actually loaded. The final site matrix is `0/3/4/5/7` nodes with `0–5/12/20/30/49` ore blocks outside the hearts.

This is a `FINAL_DESIGN` contract. Registration, assets, recipes, tests, hosted build evidence, and runtime behavior remain pending until the canonical acceptance checklist is proven.

GeOre remains a separately distributed required dependency on the supported `6.2.2` line. IOE's physical-resource suppression may still remove autonomous GeOre Overworld `*_geode` placed features so the final expedition path remains authoritative.

## Ownership and provenance

- GeOre owns its blocks, growth, drops, recipes, models, textures, sounds, translations, and upstream geode implementation.
- Immersive Engineering owns the abstract mineral deposits used by the reworked IOE distribution.
- IOE owns its suppression policy, biome mapping, Budding rank variants, acquisition recipes, site metadata, and transactional site planning.
- No GeOre Java source, data, model, texture, sound, language file, loot table, or recipe is copied into IOE.

The inspected artifact is `GeOre-1.21.1-6.2.2.jar` (`maven.modrinth:Xw6zG9hl:OV3DOSul`). See `THIRD_PARTY_NOTICES.md` for preserved MIT attribution.

## GeOre: Additions boundary

The separate GeOre: Additions restriction remains a defensive compatibility policy for its budding-harvester and extractor capabilities. It prevents those capabilities from bypassing the final acquisition rules. See `GEORE_ADDITIONS_COMPATIBILITY_PLAN.md`.
